# Prototype 2 Data Schema

This document summarises the current exported session JSON produced by the Prototype 2 app.

**Implementation note:** the JavaScript code files are the source of truth. This document is a human-readable guide for downstream extraction and QC.

## Source files

The schema is defined across:

- `schema.js` — schema/app constants and rolling-window configuration;
- `logger.js` — session object, event logging, redaction, quality summary and JSON export;
- `tasks.js` — base task definitions and banking library data;
- `app.js` — generated task sequence, generated content, task evidence and validation logic;
- `firebase.js` — anonymous Firebase identity and session upload.

## Session file naming

Manual JSON export uses:

```text
participantId_sNNN_sessionId.json
```

Example shape:

```text
pABC123_s001_0123456789abcdef.json
```

Firebase upload stores the same completed session object as JSON in Firebase Storage and writes a metadata record to Firestore.

## Top-level session object

A completed session object contains the following major top-level fields.

| Field | Meaning |
|---|---|
| `schemaName` | Schema identifier. |
| `schemaVersion` | Schema version string. |
| `appVersion` | App build/version string. |
| `appMode` | Current app mode, currently `collection`. |
| `sessionId` | Random per-session ID. |
| `sessionIndex` | Incrementing local browser session count. |
| `participantId` | Pseudonymised participant identifier. |
| `identitySource` | Whether identity came from localStorage, memory fallback or fresh minting. |
| `startedAtIso` | Session start timestamp. |
| `createdAtClientISO` | Client creation timestamp. |
| `completedAtIso` | Completion timestamp once finalised. |
| `exportedAtIso` | Manual/share export timestamp if manual export is used. |
| `exportMethod` | Manual/share export method if manual export is used. |
| `startedAtPerf` | Browser performance timer value at start. |
| `sessionDurationMs` | Duration from start to completion. |
| `completedNormally` | Boolean completion flag. |
| `context` | Self-reported context answers. |
| `generatedContent` | App-generated fake banking content and task-generation metadata. |
| `config` | Copy of `SESSION_CONFIG`. |
| `privacyFlags` | Explicit privacy flags. |
| `device` | Browser/device metadata. |
| `capabilities` | Browser/device capability metadata. |
| `permissions` | Motion/orientation/geolocation permission states. |
| `events` | Raw event stream. |
| `taskSummary` | One completed summary row per task instance. |
| `qualitySummary` | Session-level quality and missingness summary. |

## Identity fields

Participant identity uses local browser storage rather than real-world identity.

- `participantId` is a pseudonymous ID such as `pABC123`.
- `sessionIndex` increments in the same browser.
- `sessionId` is unique per session.
- If localStorage is unavailable, the logger can fall back to in-memory identity for the current browser session.

Downstream analysis should treat `participantId` as the user label and `sessionIndex` as a repeated-session/time-order indicator.

## Current context fields

The current context questionnaire records:

| Field | Meaning |
|---|---|
| `timeOfDay` | Morning, afternoon, evening or night. |
| `fatigue` | Slider value from 1 to 5. |
| `focusLevel` | Slider value from 1 to 5. |
| `devicePlatform` | iPhone, Android phone, tablet, laptop/desktop or other. |
| `deviceModel` | Specific device model/category based on selected platform. |
| `inputDevice` | Touch, trackpad, mouse, keyboard combinations or other. |
| `movement` | None, slight movement, walking/moving, public transport/vehicle or other. |
| `environmentNoise` | Quiet through very noisy. |
| `privacy` | Private, shared room, public place or other people nearby. |
| `posture` | Sitting, standing or walking/moving. |
| `handUse` | Two-handed, one-handed right or one-handed left. |

Categorical values are slugified by the frontend, so downstream code should expect machine-friendly strings rather than the exact display labels.

## Generated content

`generatedContent` stores fake app-created content and task-generation metadata. This is safe to store because it is generated by the app, not entered privately by the participant.

Important fields include:

- `taskSequenceMode`
- `totalTaskInstanceCount`
- `contentSetCount`
- `repeatedTaskIds`
- `contentSets`
- `randomization`
- `gestureTasks`

Each content set includes generated target merchant/category/amount information, transfer amount, insight target, spending-card target, approval payment and option/list counts.

Downstream extraction should use this content for audit, task validation and stratification, but should avoid treating exact generated merchant/category text as identity evidence.

## Config fields

`config` is copied from `SESSION_CONFIG` and currently includes:

- target session metadata;
- recommended rolling-window length and step;
- explicit raw-text/geolocation flags;
- participant identity method;
- gesture task list;
- event schema notes;
- high-frequency throttle settings.

The recommended continuous-authentication representation is:

```text
window length = 7500 ms
step size     = 2500 ms
```

## Privacy flags

The session includes explicit privacy flags:

- `storesRawText: false`
- `storesGeolocation: false`
- `typedTextRedacted: true`
- `userTypedContentStored: false`

These flags should be checked during QC.

## Device and capability metadata

`device` includes browser/device metadata such as:

- user agent;
- platform;
- vendor;
- language;
- max touch points;
- inferred device family;
- pointer/touch support;
- visual viewport support;
- motion/orientation support;
- hover/pointer media-query signals;
- viewport and screen dimensions;
- timezone;
- connection metadata where available.

`capabilities` includes support flags for storage, vibration, generic sensors, accelerometer, gyroscope, magnetometer and device motion/orientation permission APIs.

These fields are important for support/missingness checks and device-aware evaluation. They should be used carefully because device/browser artefacts can leak identity if a participant is uniquely associated with one device.

## Permissions

`permissions` currently tracks:

- `motion`
- `orientation`
- `geolocation`

Geolocation should remain `not_collected_privacy`. Motion and orientation can be `not_requested`, `granted`, `denied`, `not_required_or_unavailable`, `error` or browser-specific equivalents.

## Raw event schema

Every event in `events` follows this general shape:

| Field | Meaning |
|---|---|
| `kind` | Event type. |
| `t` | Duplicate/compatibility event type field. |
| `tRelMs` | Relative time since session start in ms. |
| `ms` | Duplicate/compatibility relative-time field. |
| `timestampIso` | ISO timestamp. |
| `tISO` | Duplicate/compatibility ISO timestamp field. |
| `taskId` | Current generated task instance ID. |
| `taskIndex` | Current generated task index. |
| `screenId` | Current screen identifier. |
| `activeArea` | Area currently being viewed/used. |
| `instructionArea` | Area required by the current guided task. |
| `componentId` | Component identifier if available. |
| `payload` | Event-specific sanitised payload. |

`taskId` is the generated task instance ID, for example `activity_search_01`, not always the base task ID.

## Event families

The current app can produce the following event families.

### Session and task events

- `session_start`
- `session_complete`
- `task_start`
- `task_end`
- `screen_view`
- `nav_tab_clicked`
- `visibilitychange`
- `resize`
- `orientationchange`

### Pointer and touch events

- `pointerdown`
- `pointermove`
- `pointerup`
- `pointercancel`
- `pointerrawupdate` where available
- `touchstart`
- `touchmove`
- `touchend`
- `touchcancel`

Pointer payloads can include coordinates, normalised coordinates, screen/page coordinates, movement deltas, pressure, contact width/height, tilt, twist and coalesced event count.

Touch payloads can include touch counts, changed touch counts, coordinates, normalised coordinates, force, radius, rotation angle, centroid and pinch distance.

### Input and keyboard events

- `focus`
- `blur`
- `keydown`
- `keyup`
- `beforeinput`
- `input`
- `select`
- `copy`
- `cut`
- `paste`

Input payloads store privacy-safe metadata such as:

- component ID;
- key class;
- code class;
- input type;
- value length;
- delta length;
- data length;
- selection ranges;
- modifier keys;
- repeat/composition flags;
- clipboard text length, not clipboard text.

### Scroll and viewport events

- `scroll`
- `window_scroll`
- `visual_viewport_resize`
- `visual_viewport_scroll`
- `carousel_scroll`

Scroll payloads store raw and ratio-based scroll positions, possible max values and geometry fields.

### Motion and orientation events

- `sensor_permission_result`
- `sensor_permission_error`
- `motion_logging_started`
- `devicemotion`
- `deviceorientation`

Motion/orientation may be absent depending on device, browser and permission state. Missing motion/orientation alone should not make a session unusable.

### Custom task/action events

Examples include:

- `passcode_digit_tap`
- `merchant_option_selected`
- `activity_filter_selected`
- `transaction_selected`
- `category_selected`
- `pot_selected`
- `pot_transfer_confirmed`
- `spending_card_selected`
- `card_swipe_summary`
- `insight_card_selected`
- `gesture_drag_start`
- `gesture_drag_end`
- `pot_drag_release`
- `approval_swipe_release`
- `secure_approval_selected`
- `final_feeling_selected`

These events provide useful audit and validation evidence, but downstream identity features should prioritise behavioural dynamics over exact task outcome labels.

## Redaction and sanitisation

The logger sanitises payloads before storage.

The current redaction blocks exact or substring keys associated with raw/user text, including:

- `rawText`
- `typedText`
- `inputValue`
- `userInput`
- `searchText`
- `clipboardText`
- `password`
- `passphrase`
- `email`
- `phone`

Safe generated/numeric fields such as `valueLength`, `deltaLength`, `currentValue`, `targetValue`, `releasedValue`, `currentRatio` and `releasedRatio` should remain visible.

## Task summary schema

Each completed task instance appends one row to `taskSummary`.

| Field | Meaning |
|---|---|
| `taskId` | Generated task instance ID. |
| `taskType` | Base task type, such as `typing_search` or `pot_drag`. |
| `taskArea` | Guided task area. |
| `taskIndex` | Generated task index. |
| `startedAtMs` | Relative task start time. |
| `completedAtMs` | Relative task completion time. |
| `durationMs` | Task duration. |
| `completed` | Completion flag. |
| `evidence` | Sanitised task-specific validation/evidence object. |

A normal full current session should contain 26 task summaries.

## Quality summary

`qualitySummary` includes:

- completed task count;
- expected task count;
- expected task IDs;
- completed task IDs;
- missing task IDs;
- event count;
- modality presence flags;
- warnings;
- `usableForSignalExtraction`.

Current warnings can include:

- `missing_task_summaries`
- `missing_typing_events`
- `missing_scroll_events`
- `missing_card_swipe_summary`
- `missing_approval_swipe`
- `missing_motion_events`
- `missing_orientation_events`

`usableForSignalExtraction` ignores missing motion/orientation warnings because these sensors may be unsupported or unavailable.

## Downstream extraction guidance

Recommended extraction steps:

1. Load each raw session JSON.
2. Validate top-level schema/app fields.
3. Check `privacyFlags`.
4. Check final event and completion fields.
5. Check `qualitySummary` and `taskSummary` length.
6. Flatten `events` into an event-level table.
7. Add session-level context/device fields to every event row.
8. Derive `baseTaskId` by removing the trailing `_<two digits>` suffix from task instance IDs.
9. Build 7.5s / 2.5s rolling windows by session.
10. Extract feature families for rhythm, typing, pointer/touch, scroll, drag/swipe, motion/orientation and context/support.
11. Split by session, not random overlapping windows.
12. Treat device/browser artefacts as audit/context variables unless deliberately modelling device effects.

## Leakage warnings

Avoid these common mistakes:

- Do not split overlapping windows randomly across train/test.
- Do not use generated merchant/category/note text as direct identity evidence.
- Do not let unique device/browser fields dominate identity classification without explicit analysis.
- Do not treat missing optional sensors as failure.
- Do not mix task outcome correctness with behavioural identity unless that is a deliberate analysis.
- Do not assume pointer and touch streams are independent; on some devices they may describe the same physical action.

## Current expected normal session

A normal completed session should have:

- `completedNormally: true`;
- one final `session_complete` event;
- 26 task summaries;
- input events;
- scroll events;
- `card_swipe_summary`;
- `approval_swipe_release`;
- no raw user-entered text;
- no geolocation;
- motion/orientation if supported and permission allows.
